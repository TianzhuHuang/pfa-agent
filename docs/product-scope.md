# PFA 产品范围（Product Scope）

本文档界定 PFA（个人业务 Agent）的**目标、首个 MVP 场景**与边界，后续功能与架构均不得违背此范围。

---

## 1. 产品目标

- **面向用户**：基金经理与有经验的炒股个人；支持多用户与未来商业化。
- **核心价值**：用 Agent 完成「信息收集与整理」，让用户把时间用在「决策」上；基于用户持仓与关注渠道，做**信息降噪**与**个性化摘要/预警**。

---

## 2. 首个 MVP 场景（当前 Phase 1 焦点）

**第一个交付场景**明确为：

- **「基于我的持仓进行信息降噪」**

即：

1. 用户以多种方式之一**录入持仓**（手动 JSON、截图 OCR、浏览器读取、Excel 导入，见 `config/user-profile.schema.json`）。
2. 用户配置**关注的信息渠道**（如 RSS、雪球、Twitter 等）。
3. Agent 从这些渠道拉取信息，**仅筛选与用户持仓相关**的内容，过滤掉与己无关的噪音。
4. 将「与持仓相关的少量核心变动/摘要」推送给用户，而不是把成千上万条原始信息丢给用户。

**不追求**在 MVP 阶段一次性支持所有市场、所有数据源；先围绕「持仓 + 信息降噪」做深做透，再扩展宏观、链上、多市场等。

---

## 3. 边界（Out of Scope for MVP）

- Agent **不执行交易**：无下单、撤单、转账等写操作；仅只读（见 `docs/security-architecture.md`）。
- **不**在 MVP 中承诺支持全部券商或全部渠道；优先 1 个 RSS 源 + 持仓录入 + 一种推送方式跑通闭环。
- 合规与审计、多租户计费等，放在后续阶段。

---

## 4. 与 Phase 1 的对应关系

| Phase 1 产出              | 说明 |
|---------------------------|------|
| 持仓数据模型               | `config/user-profile.schema.json`：标的、成本价、仓位、市场、录入来源（OCR/手动/浏览器/Excel） |
| 安全架构                   | `docs/security-architecture.md`：只读模式、敏感数据脱敏、隐私不外泄 |
| 首个功能原型               | `scripts/validate_portfolio.py`：校验用户输入的 JSON 持仓是否符合 Schema |
| 产品范围与 MVP             | 本文档：首个 MVP = 基于持仓的信息降噪 |

后续 Phase 2（数据层）、Phase 3（知识库与 Skills）、Phase 4（CRON 与推送）均以本 MVP 为第一目标展开。
