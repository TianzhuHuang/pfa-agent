# PFA 安全架构设计

本文档定义 PFA（个人业务 Agent）在**只读模式**下如何处理用户持仓数据，以及如何确保隐私不外泄。所有涉及持仓与券商授权的实现必须遵循本规范。

---

## 1. 敏感数据范围

以下数据属于**最高敏感级**，不得以明文形式离开用户可控环境或进入不可信第三方：

- **持仓明细**：标的、数量、成本价、仓位、所属市场等
- **券商/交易账户凭证**：登录账号、密码、双因素信息等
- **与持仓绑定的身份或资产信息**：能反推真实资产规模的数据

---

## 2. 只读模式（Agent 权限边界）

### 2.1 定义

当用户选择通过 **Agent 操作浏览器** 方式录入持仓时，Agent 仅运行在**只读模式**下：

- **允许**：在用户授权后，控制浏览器访问用户已登录的券商页面，**读取**持仓、资金、委托等**展示类数据**，并仅在用户指定范围内（如本地或用户自建服务）进行结构化存储与计算。
- **禁止**：Agent **不具备**任何**下单、撤单、转账、修改账户设置**等写操作权限。系统不提供、不暴露任何「交易执行」能力给 Agent。

### 2.2 产品侧要求

- 在**引导与文案**中明确告知用户：「Agent 仅具备**数据读取权限**，不具备下单/交易权限。」
- 在涉及「浏览器登录券商」的流程中，提供**只读模式说明**与确认步骤，降低用户心理门槛与合规风险。

### 2.3 实现侧要求

- 若使用 Playwright/Openclaw 等浏览器自动化：仅调用**读页面内容、截图、DOM/接口读取**等能力；不注入或执行任何可触发下单/交易的脚本。
- 券商凭证（若需持久化）：见下文「敏感信息脱敏与存储」。

---

## 3. 敏感信息脱敏与存储

### 3.1 券商凭证（密码等）

- **禁止**：在云端或中心化服务中存储券商**明文密码**。
- **推荐**：
  - **方案 A**：券商凭证**仅存在于用户本地**；浏览器 Agent 仅在**用户本机环境**中运行，凭证由用户在本机输入或由本机密钥库（如系统 Keychain / Vault）提供，**不经过**远程服务器。
  - **方案 B**：若必须在服务侧触发浏览器登录，采用**本地加密存储（如 Vault）**，密钥由用户掌控，且密钥与密文不同步到第三方。

### 3.2 持仓数据

- **存储位置**：优先**本地或用户自建/自控实例**；若使用云端，需明确数据归属与加密策略（如端到端或服务端加密 + 用户可控密钥）。
- **传输**：与 Agent 或分析服务交互时，使用 TLS 等加密通道；避免在日志、调试输出中打印完整持仓明细。

### 3.3 脱敏策略（日志与调试）

- 日志中不输出完整标的列表、数量、成本价；可输出**脱敏摘要**（如「共 N 条持仓，市场：A/US/HK」）。
- 调试与错误上报时，不包含可识别个人的持仓或账户信息。

---

## 4. 隐私不外泄原则

- **最小必要**：仅收集与「基于持仓的信息降噪、分析与推送」相关的数据；不索取与投资决策无关的敏感信息。
- **用户可控**：持仓与渠道配置的增删改查均由用户主动触发；Agent 不擅自将用户数据同步到未授权的第三方。
- **可审计**：机构版或后续商业化时，应对「谁在何时访问了哪些数据」有可追溯记录（见合规文档）。
