# Chat-Preview 本地功能测试报告与修改建议

**测试页面**：http://localhost:3000/chat-preview  
**角色**：产品经理视角，主流程 + UI/体验  
**日期**：2025-03

---

## 一、已修复问题（本轮修改）

### 1. 发送文字后报错：「cannot access local variable 'get_realtime_prices'」

- **原因**：`backend/api/chat.py` 中 `_generate_inner()` 内，在「交易确认」分支里再次 `from ... import get_realtime_prices`，导致 Python 将 `get_realtime_prices` 视为该函数内的局部变量；走到「普通 AI 对话」分支时未执行该 import，访问即报 UnboundLocalError。
- **修改**：删除第 221 行的重复 import，统一使用文件顶部已导入的 `get_realtime_prices`。
- **验证**：在 chat-preview 输入任意文字发送，应正常流式回复，不再出现该错误。

### 2. 上传图片过程中无法输入文字

- **原因**：输入区 `disabled={streaming || messagesLoading || attachLoading}`，上传 OCR 期间 `attachLoading=true` 导致输入框和发送按钮被禁用。
- **修改**：改为 `disabled={streaming || messagesLoading}`，**仅**在「流式回复中」和「加载历史中」禁用输入；上传/识别过程中可照常输入文字（与豆包等产品一致）。
- **验证**：选择图片后，在「识别中...」出现时仍可输入文字、点击示例问题、发送。

### 3. 截图识别失败时提示不友好 / 未识别到持仓时误导

- **原因**：  
  - 后端返回 `status !== "ok"` 或非 200 时，前端统一显示「截图识别失败：未知错误」，未展示后端返回的 `error` 文案。  
  - 识别成功但 `holdings` 为空时（非持仓截图或模型未识别出表格），仍走失败分支，文案与「接口报错」混在一起。
- **修改**：  
  - 成功且 `holdings.length > 0`：沿用原逻辑，展示「以下为截图识别到的持仓…」。  
  - 成功且 `holdings.length === 0`：展示「未识别到持仓，请上传更清晰的持仓截图或换一张图再试。」  
  - 失败：展示「截图识别失败：」+ 后端返回的 `d.error`（若有），否则「未知错误」。
- **验证**：  
  - 上传非持仓图（如风景）：应看到「未识别到持仓…」。  
  - 未配置 `DASHSCOPE_API_KEY` 或接口异常：应看到具体错误信息（如「DASHSCOPE_API_KEY 未设置…」）。

## 二、最小验证步骤（本地快速回归）

**前置**：后端 `uvicorn` 跑在 8000，前端 `npm run dev` 跑在 3000；根目录 `.env` 已配置 `DASHSCOPE_API_KEY`；至少有一条持仓（否则会先看到「先录入持仓」）。

按顺序执行即可完成本轮修复的验证：

| # | 操作 | 预期（通过标准） |
|---|------|------------------|
| 1 | 浏览器打开 http://localhost:3000/chat-preview | 进入三栏页，左侧历史、中间对话、右侧持仓 |
| 2 | 在输入框输入「今天大盘怎么样」并发送 | **不**出现「抱歉，处理时出错：cannot access local variable 'get_realtime_prices'」；出现正常流式回复 |
| 3 | 点击输入框左侧 +，选择任意一张**非持仓**图片（如风景/截图） | 先出现「识别中...」；此时**仍能在输入框里打字**、点示例问题 |
| 4 | 等待识别结束 | 出现「未识别到持仓，请上传更清晰的持仓截图或换一张图再试。」或「截图识别失败：xxx」（若未配 Key 则见 DASHSCOPE 相关文案） |

**约 2 分钟**可跑完 1～4；第 5 步需要真实持仓截图。若 2 通过即说明 `get_realtime_prices` 已修；3～4 通过即说明「上传不挡输入」+「识别结果/错误提示」已修。

## 四、仍可能出现的「上传图片无法识别」原因（排查建议）

以下情况会导致用户感知为「无法识别」，可在文档或 UI 中说明：

1. **未配置 DASHSCOPE_API_KEY**  
   - 后端返回 `error: "DASHSCOPE_API_KEY 未设置…"`。  
   - 建议：在设置页或首次使用 OCR 时提示「需配置通义千问 API Key 才能识别截图」。

2. **图片非持仓截图或格式/清晰度问题**  
   - 模型无法解析出表格或返回空列表。  
   - 已做：前端展示「未识别到持仓，请上传更清晰的持仓截图或换一张图再试。」  
   - 可选：在附件区域增加一句说明「请上传券商持仓截图，表格清晰更易识别」。

3. **网络/超时**  
   - 通义 VL 请求超时（当前 60s）。  
   - 建议：前端对 `/api/portfolio/ocr` 做较长 timeout（如 65s），失败时展示「识别超时，请稍后重试」。

4. **图片过大**  
   - 后端限制 10MB，超过返回「图片过大，请压缩后重试」。  
   - 建议：前端在选择文件后若 `file.size > 10*1024*1024` 先提示并拒绝上传，避免长时间等待后报错。

## 五、UI/交互修改建议（产品视角）

### 4.1 输入与附件

- **占位符**：当前「粘贴链接、截图或输入问题」已较好；可考虑在附件摘要条下方增加一句「发送时将作为上下文一起提交」，减少用户疑惑。
- **上传中状态**：除「识别中...」外，可为 + 按钮增加 loading 态（如旋转图标），避免用户重复点击。
- **多图**：当前仅支持单图；若后续支持多图，需约定顺序与摘要拼接方式。

### 4.2 错误与边界

- **流式出错**：若 SSE 返回 `type: chunk` 且 content 为「抱歉，处理时出错：…」，当前已展示在气泡内；建议错误类 chunk 在 UI 上略做区分（如浅红底或小图标），便于扫视。
- **无网络/后端不可用**：发送后若请求直接失败（非 SSE），当前会 set 一条「错误: xxx」的 assistant 消息；建议对 5xx/网络错误增加 toaster 或顶部 banner，避免用户忽略。

### 4.3 会话与历史

- **新建会话**：当前选「首页对话」发即视为同一全局会话；若产品希望「新开一个对话」，需在 UI 上提供「新对话」入口并在前端/后端区分 session。
- **左侧历史**：若会话很多，可考虑分页或「加载更多」；当前 all-sessions 限制 80 条，一般够用。

### 4.4 右侧持仓面板

- **行业**：当前多为空，若后端无 industry 字段，可保留占位或从 symbol 做简单映射（如 600→白酒）作为过渡。
- **本话关联**：Phase 1 无 ImpactCard 时，高亮依赖 impactCard；有 impact 后需保证后端在 done 时带上 impact，前端再算 highlightedSymbols。

## 六、后续可做的体验优化（非必须）

- **OCR 前预览**：选择图片后先展示缩略图，再点「识别」或自动开始识别，减少误选。
- **链接粘贴**：检测输入中的 URL，先调 link-preview 再填入上下文（依赖后端 Phase 3 接口）。
- **键盘快捷键**：如 Enter 发送、Shift+Enter 换行，与主流 IM 一致。
- **无障碍**：为输入框、发送、附件按钮补全 `aria-label` 与错误状态的 `aria-live`。
