# PFA 2.0 后端调研（情报流水线）

本文档记录 Chat 2.0 所需后端能力的调研方向，供后续实现参考。**不在此实现**。

---

## 1. 爬虫服务（Crawl）

**目标**：抓取用户粘贴的新闻链接或指定来源（财联社、华尔街见闻等），提取正文，便于 LLM 理解。

**可选方案**：

| 方案 | 说明 |
|------|------|
| Playwright / Puppeteer | 无头浏览器，可渲染 JS、绕过部分反爬；维护成本高 |
| 第三方 API | 若渠道提供官方或聚合 API，直接请求；需评估稳定性与合规 |
| Readability / goose3 | 从 HTML 提取正文，轻量；对复杂页面效果一般 |
| 现成服务 | 如 Mercury Parser、Diffbot 等，按量付费 |

**建议**：优先评估「链接预览 + 正文提取」的轻量方案（Readability 类），再对重点渠道（财联社、见闻）做定制或官方 API 对接。

---

## 2. OCR 引擎

**目标**：用户上传证券 App 截图后，高精度识别持仓表，并输出**结构化 JSON**（标的、数量、成本等），便于写入持仓知识库。

**可选方案**：

| 方案 | 说明 |
|------|------|
| 通义千问 VL (Qwen-VL) | 项目已用 DASHSCOPE；多模态理解强，可输出结构化字段；需设计 Prompt 与输出 Schema |
| PaddleOCR | 开源，表格识别能力好；需后处理解析为结构化数据 |
| 其他多模态模型 | GPT-4V、Claude Vision 等，按需选型 |

**建议**：在现有 `pfa/screenshot_ocr.py` 基础上，明确输出 Schema（与 `config/user-profile.schema.json` 对齐），优先用 Qwen-VL 做端到端「截图 → JSON」流水线。

---

## 3. 推理流水线（两步走）

**目标**：用户输入一条新闻（文本/链接/截图）后，先提取特征，再与持仓知识库做相关性检索，最后生成「影响评估」回复。

**建议流程**：

1. **特征提取**  
   从新闻中抽取：标的代码/名称（如 AAPL、贵州茅台）、宏观关键词（降息、地缘冲突）、行业标签等。

2. **相关性检索**  
   将提取的关键词与用户持仓知识库（RAG）匹配：  
   - 持仓列表 + 行业/板块标签作为文档  
   - 用向量检索或关键词匹配，筛出「可能受影响的持仓」

3. **LLM 推理**  
   以「新闻摘要 + 匹配到的持仓」为上下文，生成：  
   - 影响评估卡片（标题、影响标的及程度、sentiment、三句话拆解）  
   - 可选：结构化 JSON，便于前端渲染 ImpactCard 与 Deep Dive

**技术选型**：  
- 向量库：Supabase pgvector、Chroma、本地 embedding 等  
- 推理：现有 DASHSCOPE（qwen-plus / qwen-max），必要时增加「结构化输出」Prompt 与解析

---

## 4. 与现有代码的关系

- **持仓数据**：沿用 `config/user-profile.schema.json` 与 `pfa/data/store.py`（或后端 API 暴露的持仓接口）。
- **对话存储**：可复用 `backend/` 下 chat 相关 API；会话与「情绪标签」需在存储层扩展。
- **OCR**：复用/扩展 `pfa/screenshot_ocr.py`。
- **爬虫**：新建模块，与路由/API 解耦，便于单独部署或限流。

以上为调研记录，具体实现以 Phase 1/2 排期为准。
