# PFA 规划 · 大白话版

产品与目标、各阶段在做什么，每步一句话说清，便于快速回顾。

---

## 产品是啥

帮基金经理省脑子：只做**信息筛选、影响评估、风险提示**，**不给交易指令**，买卖决策永远是人来做。

---

## 第一版只做谁、只解决啥

- **只服务**：国内做主观多头/量化辅助的基金经理（小团队 3–10 人）。
- **只解决一件事**：每天信息太多，**分不清哪些真的和我的仓位有关**。
- **算成功的标准**：用户每天至少打开 1 次，且有至少 1 条推送被标成「有用」。

---

## Phase 1：先把「规矩」定好（先别急着写业务代码）

- 建好 `docs/`、`config/`，写出产品范围、安全架构、数据源说明、用户配置的 Schema。
- 安全上：敏感信息本地/加密，Agent 只能读数据、不能下单。
- 用 `.cursorrules` 规定：后面写代码都要按这些文档来。

---

## Phase 2：把「该抓的数据」抓回来（按价值排先后）

- 先抓**和持仓强相关**的（公告、财报、监管处罚等，能和用户持仓对上）。
- 再抓**高信噪比宏观**（利率、政策、重要会议，事件型不是观点型）。
- 最后才是**观点型**（雪球、Twitter、研报等，要标清楚「这是观点不是事实」）。
- 技术细节放 data-sources.md，这里只讲「为什么要抓这些」。

---

## Phase 3：让 Agent 会「想」、能「改」

- 建一个本地知识库，把 Phase 2 抓来的数据存进去、能检索。
- 把 Skills 做成**产品能力**（例如：持仓风险扫描、事件影响评估、投资风格约束），不是一堆零散 Prompt。
- 每次给结论都要**说清楚用了哪个 Skill、哪条数据**（不黑盒）。
- 用户可以说「太保守了」之类，系统把反馈写进偏好，后面分析要跟着改（纠偏）。

---

## Phase 4：把「例行」和「值得打断」分开做

- **例行**：每天盘前/盘后、每周复盘，定时生成摘要并推送。
- **打断**：只有「持仓相关重大公告 / 突破你设的阈值 / 多源一起异动」才主动推——**「只有值得打断你的时候才打断你」**。
- 接上 Telegram 或邮件等至少一种推送；需要时再接 Openclaw 做复杂网页截图等。

---

## Openclaw 的用法

只当「复杂交互工具」用（例如要登录、反爬、复杂截图时）；平时**优先用 API / RSS**，保证简单、稳、便宜。

---

## 开发怎么推进

一步一步来：Phase 1 验收过了再做 Phase 2，以此类推；每阶段结束更新对应 `docs/`；第一周小目标：**1 个 RSS 源 + 持仓能解析 + 一种推送**先跑通。
